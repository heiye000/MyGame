# 3. 相机视角分析

## 3.1 视角类型

《困兽之国》的画面更准确地说是：

> 高角度俯视地图 + 3/4 视角人物绘制。

它不是严格的等距轴测视角，也不需要使用真正的三维倾斜相机。

主要依据：

- 地面并未统一采用 2:1 菱形网格。
- 房屋、树木和人物的透视关系具有手绘调整痕迹。
- 人物正面、背面和侧面仍保留大量身体信息，并非只看见头顶。
- 场景遮挡主要通过二维绘制顺序和 Y Sort 实现。

### Godot 实现

应使用：

```text
Node2D 世界
└── Camera2D
```

不建议为了模拟俯视感使用：

```text
Camera3D + 倾斜平面 + 2D Billboard
```

后者会增加：

- 像素缩放误差；
- 角色与地面排序复杂度；
- 贴图采样模糊；
- 碰撞和导航转换成本。

---

## 3.2 逻辑分辨率

推荐：

```text
640 × 360
```

输出关系：

| 输出分辨率 | 理想整数倍率 |
|---|---:|
| `1280×720` | 2× |
| `1920×1080` | 3× |
| `2560×1440` | 4× |
| `3840×2160` | 6× |

`640×360` 并非原作公开参数，而是根据截图中的像素颗粒、人物占屏比例和 16:9 构图反推得到的适配值。

### 选择 640×360 的原因

- 人物可见高度约 36～40 个逻辑像素时，占屏高度约 10%～11%。
- 能同时展示较大的探索范围。
- 720p、1080p、1440p 均可整数放大。
- 足以容纳开放世界场景细节，不会像 `320×180` 那样过于粗糙。
- 相比 `960×540`，人物仍能保持明显的像素颗粒。

---

## 3.3 玩家在画面中的构图位置

建议默认位置：

```text
水平：50%
垂直：54%～56%
```

也就是角色位于画面中心略偏下。

对于 `640×360`：

```text
画面绝对中心：Vector2(320, 180)
建议角色显示位置：Vector2(320, 194～202)
```

因此，相机中心应位于玩家脚底上方约：

```text
16～22 个逻辑像素
```

相机目标可写成：

```text
camera_target = player_foot_position + Vector2(0, -18)
```

这样角色会显示在屏幕中心下方约 18 像素。

---

## 3.4 相机前视

《困兽之国》的镜头整体较稳定，不适合采用夸张的动态前视。

推荐值：

| 参数 | 建议值 |
|---|---:|
| 水平前视 | `8～16 px` |
| 垂直前视 | `6～12 px` |
| 前视插值速度 | `6～9` |
| 跟随插值速度 | `8～12` |
| 战斗锁定时前视 | 降低或关闭 |
| 室内场景前视 | `0～6 px` |

前视方向应取：

1. 锁定目标方向；
2. 瞄准方向；
3. 移动方向；
4. 均无时为零。

不要仅以角色朝向决定前视，否则玩家站立转身时镜头会反复移动。

---

## 3.5 像素画相机脚本

推荐将相机独立放在世界节点下，不作为玩家子节点。

```text
World
├── Player
└── WorldCamera
```

`pixel_camera.gd`：

```gdscript
class_name PixelCamera
extends Camera2D

@export var target: Node2D
@export var follow_rate: float = 10.0
@export var look_ahead_rate: float = 7.0
@export var look_ahead_distance: Vector2 = Vector2(12.0, 8.0)
@export var composition_offset: Vector2 = Vector2(0.0, -18.0)

var desired_direction: Vector2 = Vector2.ZERO
var current_look_ahead: Vector2 = Vector2.ZERO


func set_look_direction(direction: Vector2) -> void:
    if direction.length_squared() < 0.0001:
        desired_direction = Vector2.ZERO
    else:
        desired_direction = direction.normalized()


func _process(delta: float) -> void:
    if not is_instance_valid(target):
        return

    var desired_look_ahead := Vector2(
        desired_direction.x * look_ahead_distance.x,
        desired_direction.y * look_ahead_distance.y
    )

    var look_weight := 1.0 - exp(-look_ahead_rate * delta)
    current_look_ahead = current_look_ahead.lerp(
        desired_look_ahead,
        look_weight
    )

    var desired_position := (
        target.global_position
        + composition_offset
        + current_look_ahead
    )

    var follow_weight := 1.0 - exp(-follow_rate * delta)
    global_position = global_position.lerp(
        desired_position,
        follow_weight
    ).round()
```

### 参数说明

- `composition_offset.y = -18`：使玩家位于画面中心下方。
- `round()`：将最终相机位置对齐到逻辑像素。
- 指数插值不依赖固定帧率。
- 不再同时开启 `Camera2D.position_smoothing_enabled`，避免两套平滑叠加。

### 瞬移处理

玩家传送、切场景或复活时，应立即重置镜头：

```gdscript
func snap_to_target() -> void:
    if not is_instance_valid(target):
        return

    current_look_ahead = Vector2.ZERO
    global_position = (
        target.global_position
        + composition_offset
    ).round()
```

# 4. 角色像素尺寸

## 4.1 截图测量结论

根据本次提供的截图和公开画面估算：

### 720p 输出画面中的人物

```text
可见宽度：约 26～40 输出像素
可见高度：约 66～84 输出像素
```

按 2× 整数输出反推：

```text
可见宽度：约 13～20 逻辑像素
可见高度：约 33～42 逻辑像素
```

装备会显著改变宽度：

- 素体或普通服装：13～16 px；
- 轻甲、披风：16～20 px；
- 持盾、长柄武器：可能超过 24 px；
- 攻击动作的总包围盒可能达到 50～80 px。

---

## 4.2 可见身体尺寸与动画帧格

必须区分：

### 可见身体

实际有颜色像素的区域，例如：

```text
16 × 38 px
```

### 动画帧格

包含透明边距、武器、头发、斗篷和动作摆幅的矩形，例如：

```text
48 × 64 px
```

不要把角色身体直接顶满帧格，否则会出现：

- 不同方向锚点漂移；
- 武器动作被裁切；
- 斗篷和头发没有缓冲区；
- 攻击动画无法与移动动画共用原点。

---

## 4.3 推荐帧格规格

### 方案 A：统一规格，适合主角和重要 NPC

```text
移动动画帧格：48×64 px
身体可见高度：34～42 px
脚底锚点：Y = 54～58
```

优点：

- 可容纳盾牌、背包和短武器；
- 所有方向使用统一锚点；
- 后期扩展装备更方便；
- 主角不需要频繁更换 Sprite Sheet 单元尺寸。

### 方案 B：紧凑规格，适合普通 NPC

```text
移动动画帧格：32×48 px
身体可见高度：30～38 px
脚底锚点：Y = 41～44
```

缺点是持盾、披风和宽动作容易超出帧格。

### 方案 C：战斗动作

```text
普通攻击：64×64 px
大范围攻击：96×96 px
Boss 或长柄武器：按动作单独扩展
```

移动和战斗动画可以使用不同尺寸，但脚底世界锚点必须一致。

---

## 4.4 人物比例建议

若身体总高为 38 px，可使用以下分配：

| 部位 | 高度 |
|---|---:|
| 头发与头部 | 9～11 px |
| 颈肩 | 2～3 px |
| 躯干 | 11～13 px |
| 骨盆 | 3～4 px |
| 大腿与小腿 | 11～13 px |
| 脚部 | 2～3 px |

### 视觉特点

《困兽之国》式人物不是大头 Q 版，应避免：

- 头部占身体高度 1/3；
- 肩宽过窄；
- 腿部只有 5～7 px；
- 正面和背面只靠换发型区分。

更适合采用：

```text
头身比约 1:3.5～1:4.2
```

在小尺寸下，身体轮廓比五官细节更重要。

# 5. 脚底锚点、碰撞与 Y Sort

## 5.1 世界原点

角色的世界定位点应放在：

> 两脚之间、脚掌接触地面的中心。

不要把节点原点放在：

- 图片中心；
- 腰部；
- 头顶；
- 碰撞体中心。

推荐：

```text
角色场景原点 = 脚底中心
Sprite 向上偏移
```

示例：

```text
CharacterBody2D：位置为脚底
AnimatedSprite2D.position = Vector2(0, -56)
```

实际偏移取决于 Sprite 是否启用 `centered`。

更稳定的做法是：

- `AnimatedSprite2D.centered = false`
- 将每帧左上角按统一帧格导入；
- 使用 `offset` 对齐脚底。

---

## 5.2 碰撞体

开放世界 ARPG 不应让整个人物身体都参与地面阻挡。

推荐碰撞区：

```text
宽度：10～14 px
高度：6～9 px
```

位置：

```text
脚底上方 3～5 px
```

可使用：

```text
CapsuleShape2D
```

或小型矩形。

原因：

- 人物上半身应允许进入树冠、屋檐和桌面下方；
- 狭窄通道更自然；
- 角色不会因肩膀碰撞而卡住；
- Y Sort 与碰撞的空间语义一致。

---

## 5.3 阴影

建议使用独立 `Sprite2D`：

```text
尺寸：12～18 × 5～8 px
透明度：25%～45%
位置：脚底下方 1～2 px
```

影子不参与 Y Sort，始终在角色身体下方。

待机时影子基本不缩放；跳跃或击飞时才改变大小和透明度。

---

## 5.4 Y Sort

推荐节点：

```text
World
└── YSortedEntities
    ├── Player
    ├── NPC
    ├── TreeTrunk
    └── Prop
```

`YSortedEntities.y_sort_enabled = true`

所有参与排序对象的节点原点都必须代表其地面接触点。

例如：

- 人物：脚底；
- 树：树干根部；
- 桌子：桌腿与地面的中心；
- 房屋：门槛或指定遮挡基线。

# 6. 八方向系统

## 6.1 方向名称

建议统一使用：

```text
n   北
ne  东北
e   东
se  东南
s   南
sw  西南
w   西
nw  西北
```

Godot 坐标中：

```text
右：Vector2(1, 0)
下：Vector2(0, 1)
左：Vector2(-1, 0)
上：Vector2(0, -1)
```

---

## 6.2 八方向角度区间

| 方向 | 中心角度 | 大致范围 |
|---|---:|---:|
| E | 0° | -22.5°～22.5° |
| SE | 45° | 22.5°～67.5° |
| S | 90° | 67.5°～112.5° |
| SW | 135° | 112.5°～157.5° |
| W | 180° | 157.5°～-157.5° |
| NW | -135° | -157.5°～-112.5° |
| N | -90° | -112.5°～-67.5° |
| NE | -45° | -67.5°～-22.5° |

---

## 6.3 方向转换代码

`direction_8.gd`：

```gdscript
class_name Direction8
extends RefCounted

const NAMES := [
    &"e",
    &"se",
    &"s",
    &"sw",
    &"w",
    &"nw",
    &"n",
    &"ne",
]


static func from_vector(
    value: Vector2,
    fallback: StringName = &"s"
) -> StringName:
    if value.length_squared() < 0.0001:
        return fallback

    var sector := int(round(value.angle() / (PI / 4.0)))
    sector = wrapi(sector, 0, 8)
    return NAMES[sector]
```

### 验证样例

```gdscript
Direction8.from_vector(Vector2.RIGHT)      # e
Direction8.from_vector(Vector2.DOWN)       # s
Direction8.from_vector(Vector2.LEFT)       # w
Direction8.from_vector(Vector2.UP)         # n
Direction8.from_vector(Vector2(1, -1))     # ne
Direction8.from_vector(Vector2(-1, 1))     # sw
```

---

## 6.4 移动方向与朝向分离

ARPG 中至少要区分两个向量：

```text
move_direction
facing_direction
```

它们不一定相同。

示例：

- 玩家锁定北方敌人，同时向西侧移；
- 弓箭手向东北瞄准，同时向南后退；
- 攻击动画期间停止移动，但仍保持攻击朝向。

因此，建议角色持有：

```gdscript
var move_direction: Vector2 = Vector2.ZERO
var facing_direction: Vector2 = Vector2.DOWN
var facing_name: StringName = &"s"
```

更新优先级：

```text
锁定或瞄准方向 > 攻击锁定方向 > 移动方向 > 上一次朝向
```

# 7. 五方向绘制与八方向绘制

## 7.1 第一阶段：五方向绘制

实际绘制：

```text
n
ne
e
se
s
```

镜像获得：

```text
nw = ne 水平翻转
w  = e  水平翻转
sw = se 水平翻转
```

总工作量：

```text
待机：5 × 4 = 20 帧
行走：5 × 6 = 30 帧
合计：50 帧
```

适合：

- 原型阶段；
- 无固定左右手的角色；
- 身体基本对称的 NPC；
- 尚未确定装备系统时。

---

## 7.2 最终阶段：完整八方向

总工作量：

```text
待机：8 × 4 = 32 帧
行走：8 × 6 = 48 帧
合计：80 帧
```

适合：

- 主角；
- 左右手装备明显不同；
- 有单肩护甲、剑鞘、披风扣等不对称设计；
- 武器和盾牌不独立分层的角色。

---

## 7.3 镜像的风险

水平镜像会交换：

- 左手和右手；
- 武器手和盾牌手；
- 单肩护甲方向；
- 腰间剑鞘方向；
- 发型分缝；
- 单侧伤疤或饰品；
- 固定光照方向。

所以推荐策略：

| 对象 | 策略 |
|---|---|
| 普通平民 NPC | 5 方向镜像 |
| 动物 | 5 方向或更少 |
| 主角素体 | 8 方向 |
| 武器 | 独立层或 8 方向 |
| 盾牌 | 独立层或 8 方向 |
| 披风 | 8 方向优先 |
| 对称怪物 | 5 方向镜像 |
| Boss | 独立完整动画 |

# 8. 待机动画规范

## 8.1 基础参数

```text
方向数：8 个显示方向
实际绘制：5 或 8 个方向
帧数：4 帧
播放速度：5～7 FPS
单循环：约 0.65～0.9 秒
循环：开启
```

推荐先使用：

```text
4 帧，6 FPS
```

---

## 8.2 四帧动作设计

| 帧 | 动作 |
|---|---|
| 0 | 中立姿势 |
| 1 | 胸肩上升 1 px，轻微吸气 |
| 2 | 回到中立 |
| 3 | 肩膀下降或衣摆轻微变化 |

### 必须保持不动的区域

- 左右脚的地面接触点；
- 脚底锚点；
- 整个角色的世界高度；
- 影子中心。

### 可移动区域

- 胸部：0～1 px；
- 肩部：0～1 px；
- 头部：可延迟 1 帧；
- 手臂：边缘变化 1 px；
- 披风或衣摆：1～2 px；
- 武器尖端：1 px 内。

---

## 8.3 避免“漂浮式待机”

不要让整个人物按照下面方式循环：

```text
Y = 0, -1, -2, -1
```

这样会像漂浮或上下抖动。

更好的处理：

```text
脚部固定
躯干局部变化
头部和装备产生轻微滞后
```

---

## 8.4 待机变化动画

基础待机完成后，可以增加低频随机动画：


建议：

- 不影响角色世界位置；
- 播放结束回到标准待机。

# 9. 行走动画规范

## 9.1 基础参数

推荐：

```text
6 帧/方向
10 FPS
循环时长 0.6 秒
```

可调范围：

```text
9～12 FPS
```

主角若需要更顺滑，可升级为 8 帧，但不建议原型阶段直接制作八帧。

---

## 9.2 六帧步态分解

| 帧 | 动作阶段 | 重心 |
|---|---|---|
| 0 | 左脚前触地 | 稍低 |
| 1 | 左腿承重、身体下压 | 最低 |
| 2 | 双腿交叉经过 | 上升 |
| 3 | 右脚前触地 | 稍低 |
| 4 | 右腿承重、身体下压 | 最低 |
| 5 | 双腿交叉经过 | 上升 |

### 脚步事件帧

建议在：

```text
帧 0：左脚落地
帧 3：右脚落地
```

触发：

- 脚步音效；
- 尘土粒子；
- 草地扰动；
- 水面涟漪；
- 极小幅相机反馈。

---

## 9.3 身体起伏

行走时角色躯干相对脚底的变化建议：

```text
最大 1～2 px
```

不要让脚底锚点随身体起伏移动。

行走动画中的起伏来自：

- 膝盖弯曲；
- 骨盆上下变化；
- 肩膀反向摆动；
- 装备惯性。

---

## 9.4 八方向的差异

### 南、北方向

- 双腿需要有明显左右交替；
- 肩膀和手臂使用左右摆动；
- 北方向背部轮廓必须清楚；
- 不要仅把南方向换成后脑勺。

### 东、西方向

- 步幅最容易表现；
- 膝盖前后关系要明确；
- 身体通常不是完全 90° 侧面，可保留少量 3/4 角度。

### 斜方向

- 是体现品质的重点；
- 身体宽度介于正面和侧面之间；
- 前侧手臂与后侧手臂要有遮挡；
- 远侧腿颜色可以略暗；
- 不能只是把正面动画横向压缩。

---

## 9.5 行走速度与动画速度

建议按身体高度标定移动速度，而不是盲目复制像素值。

公式：

```text
走速 ≈ 身体高度 × 1.8～2.2 / 秒
跑速 ≈ 身体高度 × 2.6～3.2 / 秒
```

若可见身体高为 38 px：

```text
走速：68～84 px/s
跑速：99～122 px/s
```

第一版推荐：

```text
walk_speed = 74 px/s
run_speed  = 108 px/s
```

---

## 9.6 动画速度随移动速度变化

基准：

```text
walk_animation_fps = 10
walk_speed = 74 px/s
```

实际播放倍率：

```text
speed_scale = 实际移动速度 / 基准移动速度
```

限制范围：

```text
0.75～1.35
```

避免减速状态下动画几乎停止，或加速状态下腿部抽搐。

# 13. 方向动画控制器

`directional_animator.gd`：

```gdscript
class_name DirectionalAnimator
extends Node

@export var sprite: AnimatedSprite2D
@export var use_horizontal_mirroring: bool = false
@export var idle_state: StringName = &"idle"
@export var walk_state: StringName = &"walk"
@export var movement_dead_zone: float = 0.12
@export var minimum_walk_speed: float = 1.0

var facing_name: StringName = &"s"
var current_state: StringName = &"idle"


func update_locomotion(
    input_direction: Vector2,
    actual_speed: float,
    base_walk_speed: float
) -> void:
    if input_direction.length() >= movement_dead_zone:
        facing_name = Direction8.from_vector(
            input_direction,
            facing_name
        )

    if actual_speed >= minimum_walk_speed:
        current_state = walk_state
    else:
        current_state = idle_state

    _play_current_animation(actual_speed, base_walk_speed)


func force_facing(direction: Vector2) -> void:
    if direction.length_squared() < 0.0001:
        return

    facing_name = Direction8.from_vector(
        direction,
        facing_name
    )


func force_state(state_name: StringName) -> void:
    current_state = state_name
    _play_current_animation(0.0, 1.0)


func _play_current_animation(
    actual_speed: float,
    base_walk_speed: float
) -> void:
    if not is_instance_valid(sprite):
        return

    var source_direction := facing_name
    var should_flip := false

    if use_horizontal_mirroring:
        match facing_name:
            &"w":
                source_direction = &"e"
                should_flip = true
            &"nw":
                source_direction = &"ne"
                should_flip = true
            &"sw":
                source_direction = &"se"
                should_flip = true

    sprite.flip_h = should_flip

    var animation_name := StringName(
        "%s_%s" % [current_state, source_direction]
    )

    if not sprite.sprite_frames.has_animation(animation_name):
        push_warning(
            "Missing animation: %s" % animation_name
        )
        return

    if sprite.animation != animation_name:
        sprite.play(animation_name)

    if current_state == walk_state and base_walk_speed > 0.0:
        sprite.speed_scale = clamp(
            actual_speed / base_walk_speed,
            0.75,
            1.35
        )
    else:
        sprite.speed_scale = 1.0
```

# 14. 玩家移动示例

`player.gd`：

```gdscript
class_name Player
extends CharacterBody2D

@export var walk_speed: float = 74.0

@onready var animator: DirectionalAnimator = (
    $DirectionalAnimator
)


func _physics_process(_delta: float) -> void:
    var input_direction := Input.get_vector(
        "move_left",
        "move_right",
        "move_up",
        "move_down"
    )

    velocity = input_direction * walk_speed
    move_and_slide()

    animator.update_locomotion(
        input_direction,
        velocity.length(),
        walk_speed
    )
```

`Input.get_vector()` 会自动限制向量长度，因此斜向移动不会变成水平速度的约 1.414 倍。

# 15. 与 LimboAI / HSM 的衔接

建议职责划分：

```text
LimboHSM：决定角色现在处于什么游戏状态
DirectionalAnimator：决定该状态应播放哪个方向动画
AnimationPlayer 或帧事件：负责精确触发攻击判定
```

状态示例：

```text
IdleState
MoveState
AttackState
RollState
HurtState
DeathState
```

### IdleState

```text
进入：
- animator.force_state("idle")

更新：
- 若有移动输入，派发 move 事件
```

### MoveState

```text
进入：
- animator.force_state("walk")

更新：
- 根据输入更新 velocity
- animator.update_locomotion(...)
- 输入归零时回到 idle
```

### AttackState

```text
进入：
- 锁定 facing_name
- animator.force_state("attack_light_1")
- 禁止普通移动状态修改动画

结束：
- 由动画完成事件派发 EVENT_FINISHED
```

### 关键原则

不要让以下系统同时控制同一个动画：

- `LimboState`
- `AnimationTree`
- 玩家移动脚本
- 武器脚本

应只有一个动画入口，例如：

```text
DirectionalAnimator
```

其他状态系统只提出请求。

# 16. 像素渲染设置

以下按 Godot 4.x / 4.7 的常见设置名称编写，具体菜单文字可能略有差异。

## 16.1 Display > Window

```text
Viewport Width：640
Viewport Height：360
Stretch Mode：viewport
Stretch Aspect：expand 或 keep
Scale Mode：integer
```

选择建议：

### `keep`

适合：

- 固定 16:9 构图；
- 不希望不同显示器展示更多地图；
- 可以接受黑边。

### `expand`

适合：

- PC 开放世界；
- 希望适配 16:10、21:9；
- 接受宽屏看到更多横向区域。

本项目更推荐：

```text
expand
```

但战斗房间、剧情镜头要额外设置 Camera Limit。

## 16.2 纹理设置

项目默认：

```text
Texture Filter：Nearest
Texture Repeat：Disabled
```

像素角色 PNG：

```text
Compression：Lossless
Mipmaps：关闭
```

对于始终以 1:1 逻辑像素显示的 2D 角色，不需要 Mipmap。

---

## 16.3 像素吸附

优先启用：

```text
rendering/2d/snap/snap_2d_transforms_to_pixel
```

不建议同时开启：

```text
snap_2d_transforms_to_pixel
snap_2d_vertices_to_pixel
```

两者同时使用可能让移动显得更不流畅。

### 建议测试两种方案

#### 方案一：严格像素清晰

```text
Transform Snap：开
Camera 最终位置 round()
VisualRoot 最终位置 round()
```

优点：

- 画面最锐利；
- 树干、角色边缘稳定。

缺点：

- 低速移动时可见 1 px 跳动。

#### 方案二：平滑移动

```text
Transform Snap：关
Camera 平滑移动
Nearest Filter
整数视口缩放
```

优点：

- 移动更顺滑。

缺点：

- 某些帧可能出现像素轻微抖动。

对于《困兽之国》式颗粒较强的像素画面，建议先采用方案一。

# 17. 视觉节点像素对齐

物理位置可以保留浮点数，视觉位置单独取整。

示例：

```gdscript
@onready var visual_root: Node2D = $VisualRoot


func _process(_delta: float) -> void:
    visual_root.global_position = global_position.round()
```

但要注意：

- `VisualRoot` 不能再继承重复位移；
- 更稳妥的方式是将物理根节点和视觉代理分离；
- 相机和角色必须使用同一套像素对齐策略；
- 不要只有角色取整而地图不取整。

第一版可以先只对相机取整，观察是否出现角色抖动，再决定是否对视觉节点取整。

# 19. 帧事件

仅靠“第几秒”触发脚步和攻击判定容易受 `speed_scale` 影响。

推荐为关键帧建立事件：

```text
walk frame 0：footstep_left
walk frame 3：footstep_right
attack frame 2：hitbox_on
attack frame 4：hitbox_off
```

实现方式：

- `AnimationPlayer` Call Method Track；
- 自定义帧事件表；
- 监听 `AnimatedSprite2D.frame_changed`；
- 攻击动画单独交给 `AnimationPlayer`。

行走脚步可用：

```gdscript
func _on_body_frame_changed() -> void:
    if $VisualRoot/Body.animation.begins_with("walk_"):
        if $VisualRoot/Body.frame == 0:
            _emit_footstep(true)
        elif $VisualRoot/Body.frame == 3:
            _emit_footstep(false)
```

需要记录上次帧，防止同一帧重复触发。

# 22. 最小可行制作顺序

## 阶段 1：相机与比例灰盒

1. 设置 `640×360`。
2. 创建 38 px 高的占位角色。
3. 相机纵向偏移设为 `-18 px`。
4. 创建树、墙、桌子测试 Y Sort。
5. 用 74 px/s 测试移动范围。
6. 在 720p、1080p、1440p 检查整数缩放。

验收标准：

- 人物占屏不显得过大；
- 画面前方有足够探索空间；
- 移动时没有明显纹理模糊；
- 树和人物遮挡关系正确。

---

## 阶段 2：五方向基础动画

制作：

```text
5 方向待机 × 4 帧
5 方向行走 × 6 帧
```

合计：

```text
50 帧
```

先不制作：

- 攻击；
- 翻滚；
- 受击；
- 装备换装；
- 披风物理；
- 随机待机。

验收标准：

- 8 个显示方向切换正确；
- 水平镜像没有锚点漂移；
- 斜向移动速度与直线一致；
- 停止移动后保持最后朝向；
- 行走时脚底没有滑动。

---

## 阶段 3：主角完整八方向

当五方向原型验证通过后：

1. 补画 `w、nw、sw`。
2. 固定左右手装备。
3. 增加头发、武器、盾牌层。
4. 确定各方向层级。
5. 添加脚步事件。
6. 最后再进入攻击和翻滚动画。

- [ ] 镜像方向没有错误交换关键装备。

## 待机

- [ ] 4 帧循环。
- [ ] 脚部固定。
- [ ] 身体局部起伏不超过 1～2 px。
- [ ] 没有整体上下漂浮。
- [ ] 所有方向的脚底线一致。

## 行走

- [ ] 6 帧循环。
- [ ] 约 10 FPS。
- [ ] 帧 0 和帧 3 对应左右脚落地。
- [ ] 行走速度与脚步节奏匹配。
- [ ] 停止时不会突然跳到错误朝向。
- [ ] 斜方向不是简单压缩正面图。
